# Plan 040: Interoperability apparatus corrective pass

## Objective

Make the Plan 038 Ubuntu interoperability apparatus executable and trustworthy before attempting protocol interoperability. This plan repairs deterministic first-run failures in source pinning, cache lookup, namespace creation, firewall policy, implementation-specific configuration, evidence retention, and cleanup ownership.

This plan is infrastructure-only. It may start one reference router at a time for readiness and RouterInfo-generation smoke tests, but it must not claim authenticated mixed-router interoperability and must not replace the explicit `blocked_missing_driver` result for non-smoke i2pr scenarios.

## Prerequisites

- Plan 039 is the controlling roadmap.
- Preserve the Plan 038 isolation and non-production boundaries.
- Use Ubuntu 24.04 amd64 as the sole supported execution target for this pass.
- Do not use distro-packaged Java I2P or i2pd as evidence.
- Do not permit public-network access during router execution.

## Deliverable 1: Canonical full source pins

Resolve the abbreviated revisions currently recorded for Java I2P and i2pd to unique 40-character Git object IDs using the locked upstream repositories.

Update `tests/integration/ntcp2/references.lock.toml` so every source revision used for checkout or evidence is a full object ID. Retain release labels as human-readable metadata, but do not use release names or abbreviated hashes as cache identity.

Update `scripts/interop/lib/common.sh` and both builders so:

- checkout detaches at the full object ID;
- `git rev-parse HEAD` must equal that full object ID exactly;
- the remote URL is verified against the lock;
- dirty source trees are rejected;
- cache keys include the full revision, lock digest, host contract, and build-command version;
- build metadata records the full revision;
- evidence accepts only full revisions;
- tests reject abbreviated revisions.

Do not resolve a revision dynamically during an offline build. Resolution is a reviewed lock-file change, not runtime behavior.

## Deliverable 2: Canonical reference identifiers and metadata parser

Select one machine identifier for each reference:

```text
java_i2p
i2pd
```

Use these exact identifiers in:

- lock tables;
- cache directory names;
- build metadata;
- CLI choices;
- scenario files;
- evidence records;
- runner lookup;
- tests and documentation.

Create a focused metadata parser in the Python harness rather than repeatedly scanning arbitrary text files. It must:

- parse an exact schema version;
- require reference identifier, full source revision, artifact hash, installed-tree hash, launcher path, host contract, and build-command version;
- reject duplicate keys and unknown required-shape drift;
- validate SHA-256 syntax;
- confirm the metadata reference matches the selected scenario reference;
- re-hash the runtime tree before launch;
- reject launcher paths that escape the cache root.

The cache resolver must not recursively pick the first metadata file with a matching substring. Resolve by canonical reference and cache key, with an explicit current-cache summary generated by `build-references.sh`.

## Deliverable 3: Strengthen the Ubuntu host contract

Update `scripts/interop/ubuntu/check-host.sh` to enforce, not merely report:

- `ID=ubuntu`;
- `VERSION_ID=24.04`;
- amd64/x86_64;
- Bash 4 or newer;
- UTF-8 locale;
- required free disk space;
- writable target directories;
- required commands and minimum tool families;
- functioning noninteractive privilege escalation when not root;
- functioning network namespace creation and deletion;
- functioning namespace-local nftables application;
- no stale Plan 038/039 namespaces, veths, or router processes.

Emit machine-readable host metadata to a caller-selected file as well as stdout. Include kernel, architecture, distribution version, Java, Ant, CMake, compiler, Python, nft, and iproute2 versions.

Update `setup-host.sh` so it runs `check-host.sh --pre-install` before `apt-get update`. The pre-install mode must perform every safe check available before modifying the host. The post-install mode must perform namespace and nftables probes and leave no residual state on any failure path.

Do not modify global firewall policy, enable services, or start routers during host setup.

## Deliverable 4: Correct and verify reference builders

### Java I2P builder

Correct the Java builder so it:

- uses the full locked source object ID;
- verifies the exact IzPack artifact before execution;
- records the exact Java, Ant, gettext, and IzPack versions;
- proves that the selected build target is the reviewed target from the pinned source;
- stages the runtime into a disposable prefix;
- discovers the approved headless launcher relative to the staged tree;
- records a normalized relative launcher path;
- performs a version or non-starting inspection probe that cannot launch a persistent router;
- checks for unexpected router processes after the probe;
- computes artifact and complete runtime-tree hashes after staging;
- rejects an existing cache if the metadata or tree hash does not match;
- makes the completed cache read-only for ordinary harness execution where practical.

If the current launcher `--help` behavior starts or attempts to start a router, replace it with a safe inspection mechanism derived from the pinned source. Do not ignore the exit status and call the probe successful.

### i2pd builder

Correct the i2pd builder so it:

- uses the full locked source object ID;
- records compiler, CMake, Boost, OpenSSL, and zlib versions;
- uses the reviewed CMake flags with UPnP disabled;
- detects and executes the actual test target at the pinned revision when available;
- verifies that `--version` reports the expected release and source revision semantics;
- treats a version mismatch as failure;
- copies all runtime-required data files into the cache;
- computes and verifies artifact and runtime-tree hashes;
- makes the completed cache read-only where practical.

### Aggregate builder

Replace ad hoc text summaries with a structured summary, preferably canonical JSON, containing:

- schema version;
- host contract digest;
- lock digest;
- canonical reference identifiers;
- full source revisions;
- cache keys;
- artifact hashes;
- installed-tree hashes;
- metadata paths relative to the repository;
- build versus cache-reuse disposition.

`build-references.sh --offline` must fail before any network operation if a required source, verified dependency, or valid cache is absent.

## Deliverable 5: Correct namespace naming and topology verification

Keep descriptive namespace names, but introduce a separate short topology token. Derive it from a collision-resistant digest of the run ID and verify uniqueness against existing interfaces and namespaces.

All generated interface names must be at most 15 bytes. Add a unit test for the maximum length and collision behavior.

Strengthen `NamespaceTopology` and `verify-isolation.sh` so they verify:

- exactly two interfaces per namespace: loopback and the expected peer interface;
- exact expected local addresses;
- exact directly connected routes;
- no IPv4 or IPv6 default route;
- public route probes fail;
- the host namespace retains no endpoint in the scenario subnet;
- forwarding is not used to create a path out of the namespace pair;
- no router process exists before verification;
- nftables rules are loaded and match the expected canonical ruleset digest.

A topology creation failure must invoke cleanup using the names already successfully created, even if later steps did not complete.

## Deliverable 6: Correct namespace-local firewall semantics

Generate separate exact rules for each namespace and address family.

For TCP:

- output permits only the exact peer destination address and exact peer listening destination port;
- input permits only the exact peer source address and the local listening destination port;
- established/related traffic is allowed;
- loopback is allowed;
- all other input and output are dropped.

Do not match the peer listening port as an inbound source port. Do not permit an entire IPv6 prefix without protocol and destination-port constraints.

Add a firewall self-test before router launch:

- permitted peer-port TCP connectivity succeeds using a disposable canary listener;
- a non-permitted peer port fails;
- a public address route fails;
- the canary is stopped and joined before router launch.

The canary must not use DNS or public endpoints.

## Deliverable 7: Correct implementation-specific configuration and paths

### Address rendering

The adapter running in the reference namespace must bind and publish the reference-side address. For the current topology this is the `.2` IPv4 endpoint, not the i2pr-side `.1` endpoint. Render IPv6 using the exact local reference address when IPv6 is enabled.

Do not duplicate literal addresses across adapters. Pass a typed topology description containing local address, peer address, local port, peer port, address family, namespace, and network ID.

### Java I2P layout proof

Using the pinned source and staged runtime, determine and document:

- base directory;
- configuration directory;
- data directory if distinct;
- RouterInfo output path and filename;
- NetDB import directory and required RouterInfo filename convention;
- headless foreground launcher semantics;
- readiness signal;
- shutdown semantics.

Update the Java adapter and adjacent README with exact source provenance. Add tests that render into a temporary run root and reject any path escaping that root.

### i2pd layout proof

Determine and document the exact pinned behavior for:

- data directory;
- RouterInfo path and filename;
- NetDB import directory and filename convention;
- network-ID configuration;
- readiness signal;
- foreground shutdown behavior.

Do not assume that copying an arbitrary RouterInfo filename into `netdb/` is sufficient. Implement the naming/placement required by the pinned source.

### IPv6 behavior

Add family-aware templates and adapter assertions. If a reference or host capability is unavailable, emit `skipped_ipv6` only after a typed capability probe. The scenario record must state the probe result and must not report a passed IPv6 scenario.

## Deliverable 8: Redesign evidence finalization

Create a dedicated evidence output root:

```text
target/interop/evidence/
```

Secret-bearing execution state remains under:

```text
target/interop/runs/<run-id>/
```

Passed and retained failed records must be written only to the evidence root. Never retain the run root to retain evidence.

Replace placeholder fields with values loaded or measured during execution:

- actual i2pr commit from `git rev-parse HEAD` with a clean/dirty disposition;
- full reference source revision;
- artifact hash;
- installed-tree hash;
- rendered configuration hash;
- canonical topology hash;
- process counters;
- namespace cleanup counters;
- real scenario and deterministic-parameter values.

Read a stored digest as a digest; do not hash the text representation of a digest again unless the schema explicitly calls for that operation.

The evidence writer must refuse `passed` if any required field is zero-filled, `record-at-execution`, missing, malformed, or inconsistent with build metadata.

Finalization order:

1. stop and join child processes;
2. collect bounded snapshots;
3. remove namespaces/interfaces;
4. generate and validate the sanitized record in memory;
5. atomically write it to `target/interop/evidence/`;
6. recursively delete the run root;
7. verify deletion and absence of processes/namespaces/interfaces;
8. rewrite/finalize the cleanup result and evidence digest atomically;
9. return success only if all previous steps pass.

Add tests proving that a passed result survives while secret files, raw logs, RouterInfo files, and run directories do not.

## Deliverable 9: Align process supervision and emergency cleanup

Choose one explicit ownership strategy:

- atomic PID files written by `BoundedProcess`, or
- namespace PID discovery using `ip netns pids` plus tracked direct child process handles.

The preferred design is to retain direct child handles for normal cleanup and use namespace PID discovery only for emergency recovery.

`cleanup.sh` must:

- validate that it operates only under `target/interop` and on expected namespace prefixes;
- enumerate namespace PIDs;
- send graceful termination;
- wait a bounded interval;
- send SIGKILL if required;
- delete namespaces;
- remove any host-side residual veth;
- delete secret-bearing run roots;
- report started, terminated, forced, namespace-deleted, interface-deleted, and failure counters;
- return nonzero if residual state remains.

A best-effort command followed by an unconditional `completed` record is insufficient.

## Deliverable 10: Regression coverage

Add focused tests for at least:

- full revision pins and exact verification;
- canonical reference identifiers;
- cache metadata parsing and path confinement;
- veth name length and collision behavior;
- correct local/peer address assignment;
- input firewall destination-port semantics;
- narrow IPv6 firewall rules;
- Java and i2pd template safety assertions;
- evidence rejection of placeholders;
- sanitized evidence survival with complete run-root deletion;
- cleanup failure overriding protocol success;
- offline builder failure before network access when cache inputs are absent;
- post-install probe cleanup on injected failure.

Use unit tests for pure generation and parser behavior. Use opt-in privileged tests for namespace/nftables behavior. Privileged tests must identify themselves clearly and remain out of ordinary unprivileged CI.

## Required validation

Run the ordinary repository quality suite and the harness unit suite. On the authorized Ubuntu host, run:

```bash

bash scripts/interop/build-references.sh --force-rebuild
bash scripts/interop/build-references.sh --offline
sudo -E bash scripts/interop/run-matrix.sh --profile environment-smoke
python3 scripts/interop/validate-evidence.py
sudo -E bash scripts/interop/cleanup.sh
```

The environment-smoke profile must prove for each reference:

- valid cache metadata and tree hash;
- namespace creation and verified isolation;
- safe rendered configuration;
- router process start and readiness;
- RouterInfo production at the proven implementation-specific path;
- bounded stop;
- sanitized evidence persistence;
- complete removal of the run root and namespace state.

It must not claim an authenticated peer connection.

## Exit criteria

Plan 040 is complete when:

- both exact reference revisions build on Ubuntu 24.04 amd64;
- offline cache reuse succeeds without network access;
- environment smoke passes for Java I2P and i2pd;
- all confirmed deterministic apparatus defects have regression tests;
- no passed evidence contains placeholders;
- no evidence-retention option preserves secrets or raw logs;
- cleanup verification reports zero residual state;
- documentation reflects the actual proven paths and commands;
- Plan 041 can begin without known build, cache, namespace, firewall, path, or evidence defects masking protocol behavior.

Do not mark Plan 038 or Milestone 3 closed after this pass. The result is an executable apparatus, not interoperability evidence.
