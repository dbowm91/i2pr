# I2P Network Protocol (I2NP)

Status: **required**  
Primary roadmap milestones: **1, 3–6**

## Scope

I2NP is the router-to-router message layer above NTCP2 and SSU2. It carries NetDB operations, tunnel construction, tunnel data, garlic/client data, delivery acknowledgements and related control traffic. Transport implementations deliver complete I2NP messages; I2NP is not a stream protocol.

## Authoritative sources

- [I2NP specification](https://i2p.net/en/docs/specs/i2np/), pinned in [SOURCES.md](../SOURCES.md), updated 2026-03 and accurate for I2NP API 0.9.69.
- [Common structures](https://i2p.net/en/docs/specs/common-structures/).
- Current NetDB, tunnel-creation and ECIES specifications referenced by each message type.

The `router.version` RouterInfo property is an I2NP feature/API version, not necessarily the implementation’s release version. Advertising a value implies the required features listed by the official I2NP version table. `i2pr` must not copy another router’s release string into this field.

## Required MVP message surface

Milestone 1 must implement the common header/envelope and typed dispatch without pretending every type is operational. Later milestones activate the messages they consume.

### Foundation

- standard and transport-specific I2NP framing forms as specified;
- message type, message ID, expiration, payload length and checksum handling;
- bounded typed payload dispatch;
- exact preservation where a message must be forwarded without semantic re-encoding;
- expiration and duplicate-detection hooks independent of transport.

### NetDB and publication

- `DatabaseStore`;
- `DatabaseLookup`;
- `DatabaseSearchReply`;
- delivery-status messages required to confirm publication or tunnel operation;
- handling of RouterInfo and supported LeaseSet-family payload types.

### Tunnels

- variable and short tunnel-build request/reply messages required by current ECIES routers;
- tunnel data and tunnel gateway messages;
- tunnel-test messages required by the advertised I2NP feature level;
- compatibility handling for long/mixed build records only if required by selected peers.

### Destination and garlic routing

- garlic messages;
- data messages and delivery instructions used for destination traffic;
- message forms required for encrypted NetDB replies and ECIES router/destination communication.

Each message type must be activated only when its downstream service exists. A decoder may recognize a type before the router can route/process it, but runtime behavior must return a typed unsupported outcome or safely discard it according to the protocol context.

## Validation and routing rules

I2NP processing must enforce:

- exact payload-size bounds before allocation;
- checksum and declared-length validation where present;
- expiration policy with an explicit clock-skew allowance;
- duplicate suppression keyed by message ID and bounded by time/memory;
- per-peer and global ingress budgets;
- no trust in transport authentication beyond peer identity and link integrity;
- typed routing outcomes: local consume, direct peer, inbound tunnel, outbound tunnel, garlic/client subsystem, reject/drop;
- no recursive immediate dispatch that permits attacker-controlled stack growth;
- bounded fan-out for replies, cloves and NetDB messages.

Transport delivery order is not guaranteed across all transports. I2NP consumers must not depend on ordered arrival unless their own protocol establishes ordering.

## Version and capability policy

Before publishing an I2NP API version, enumerate every feature that the official table makes mandatory. The pinned 0.9.69 document includes recent requirements such as tunnel testing and tunnel-build bandwidth parameters. The initial `i2pr` publication level should be the lowest truthful current value that peers will accept for the implemented roles.

Maintain separate values for:

- implementation/package version;
- I2NP feature/API version in RouterInfo;
- supported message types/features;
- transport protocol support;
- local experimental feature flags.

## Implementation references

- Java I2P: `router/java/src/net/i2p/data/i2np` plus inbound/outbound message pools and handlers.
- I2P+: corresponding I2NP classes and router message handling; compare validation and scheduling changes.
- i2pd: `libi2pd/I2NPProtocol.h` and implementation, plus NetDB/tunnel/garlic consumers.
- Emissary/go-i2p: `lib/i2np/i2np.go`, garlic-session and build-record crypto sources.

Compare message-size constants, expiration tolerances, duplicate filters, unknown-type behavior, checksum errors, forwarding without reserialization and queue priorities. Priority scheduling in another router is policy evidence, not part of the wire format unless specified.

## Required tests

- Golden envelope and payload vectors per supported message type.
- Header truncation, invalid type, invalid size, checksum failure and trailing-byte vectors.
- Expired and excessive-future messages under deterministic time.
- Duplicate delivery over the same and different transports.
- Message-ID collision/filter saturation behavior.
- Unknown but well-framed message type handling.
- Queue saturation and per-peer fairness tests.
- Forwarded-message byte preservation tests.
- Cross-implementation decode/encode fixtures for Database, tunnel-build, tunnel-data, garlic and delivery-status messages.
- Fuzzing of the envelope and every activated typed payload.

## Deferred/legacy behavior

- Legacy message types used only by NTCP1 or SSU1: legacy-reject unless current I2NP still requires safe parsing.
- Long tunnel-build formats: compatibility-only after Milestone 5 research; current ECIES short records are preferred.
- LeaseSet formats not selected in Milestones 4/6: recognize type and reject without partial trust.
- Experimental or proposed messages absent from the current official specification: experimental and non-advertised.

## Open decisions

1. Exact initial I2NP API version and the peer-eligibility consequences of that choice.
2. Whether message payloads use borrowed bounded buffers, owned canonical types, or preserved raw-plus-parsed views.
3. Duplicate-filter algorithm, epoch rotation and memory budget.
4. Priority classes and starvation guarantees across direct, NetDB, build and tunnel-data traffic.
5. How unsupported message types are surfaced to transports and metrics without creating fingerprintable or reflective error behavior.
