# SSU2 transport and reachability

Status: **required**  
Primary roadmap milestone: **8**  
Role: current UDP router transport, reachability testing and relay

## Scope

SSU2 provides authenticated point-to-point transport of I2NP messages over UDP. It includes handshake and data phases, packet-number and acknowledgement processing, loss recovery, I2NP fragmentation/reassembly, address validation/tokens, relay for NAT traversal, peer testing and connection migration.

SSU2 is semi-reliable and message-oriented. It does not provide an ordered byte stream or guaranteed delivery. I2NP messages may arrive out of order, may be delivered more than once, or may fail while the session remains usable.

## Required MVP behavior

### Address publication and socket model

- Parse/publish current SSU2 RouterAddress options, static key, introducer/relay information and reachability fields.
- Use separate IPv4 and IPv6 socket/listener state where needed for correct address and migration handling.
- Keep observed external addresses and publication decisions outside the packet codec.
- Do not publish a directly reachable address until peer-test/observation policy supports it.

### Admission, tokens and handshake

Implement explicit initiator/responder states for:

- session request/created/confirmed messages;
- retry/address-validation token issuance and validation;
- fragmented RouterInfo in establishment where required;
- transcript/KDF/header protection and authenticated payloads;
- timestamp, network-ID, peer identity and source-address checks;
- replay/duplicate establishment handling.

Token validation must occur before avoidable expensive cryptographic work. Token keys require rotation, bounded overlap and restart semantics. Invalid/spoofed-source traffic must not create unbounded per-source state.

### Data phase

Implement:

- packet header parsing, obfuscation/protection and connection IDs;
- packet-number reconstruction and replay window;
- authenticated block parsing;
- ACK ranges/NACK or loss inference as specified;
- retransmission of I2NP fragments without promising ordered I2NP delivery;
- I2NP message fragmentation/reassembly with strict budgets;
- immediate-ACK and termination behavior;
- session key rotation/rekeying;
- idle timeout and clean teardown;
- duplicate I2NP suppression at the I2NP layer in addition to packet replay checks.

Loss recovery and congestion behavior must be bounded and stable under severe loss. Avoid one task or timer per packet.

### Relay and peer test

Implement current relay roles and messages required for ordinary firewalled-router operation:

- introducer advertisement/selection;
- relay request/response/introduction;
- signed data verification and freshness;
- per-peer/IP quotas and anti-amplification checks;
- peer testing for inbound reachability and address agreement;
- state transitions that cannot be confused across concurrent tests.

A relay or peer-test response must not cause address publication from a single unauthenticated observation.

### Migration/path validation

Support source-address changes only through specification-defined validation. Do not migrate an established peer merely because an authenticated-looking packet arrives from a new address without the required path challenge/response or equivalent state.

## Security and resource requirements

- Bound datagram size before parsing and drop impossible packets cheaply.
- Separate stateless/low-state prevalidation from allocated session state.
- Rate-limit token generation, Retry, relay and peer-test responses by source and globally.
- Prevent amplification by enforcing response-size and validation rules.
- Bound active sessions, pending handshakes, packet histories, ACK ranges, retransmissions and reassembly bytes.
- Reject packet-number replays and excessively old/future windows.
- Use constant-time authentication through reviewed crypto libraries.
- Avoid reflecting detailed parse/authentication failures.
- Release packet buffers and resource permits on cancellation, migration failure and timeout.

## Required tests

- Fixed handshake/KDF/header-protection vectors for initiator and responder.
- Interoperability in both directions with Java I2P and i2pd over IPv4; IPv6 where infrastructure permits.
- Spoofed-source request floods proving bounded crypto and state.
- Token valid/invalid/expired/wrong-address/key-rotation/restart cases.
- Packet-number reconstruction, duplicate, replay-window and wrap boundaries.
- ACK ranges, loss/retransmission, severe reordering and duplicate datagrams.
- I2NP fragment loss/reorder/duplicate/conflict and reassembly pressure.
- Key rotation and termination.
- Relay requester, introducer and target roles with invalid signatures/freshness.
- Concurrent peer tests, contradictory observations and reachability transitions.
- Migration/path validation with spoofed and legitimate address changes.
- NAT-like deterministic simulation, queue saturation and cancellation.
- Fuzzing of authenticated plaintext blocks, ACKs and fragment structures.

## Deferred and excluded behavior

- SSU1: legacy-reject; the MVP excludes SSU1.
- Shared SSU1/SSU2 compatibility paths: deferred unless unavoidable for current address parsing.
- IPv6 publication without end-to-end test infrastructure: may be `required-later`, but parser/socket architecture must not preclude it.
- Exotic migration/multihoming behavior: deferred beyond specification-required ordinary address changes.
- Experimental flow-control extensions: deferred.
- Public introducer service: disabled until controlled resource/abuse testing passes even if relay-client support is complete.

## Open decisions

1. Reviewed Rust data structures for packet replay windows, ACK ranges and retransmission scheduling.
2. Token-key persistence/rotation and stateless validation format exactly matching the specification.
3. Congestion/loss-control policy where the specification permits implementation choice.
4. Per-IP/subnet admission controls that do not punish large NAT populations excessively.
5. Criteria and quorum for external-address/reachability publication.
6. Whether IPv6 is required at initial SSU2 milestone exit or may follow immediately afterward.
7. Shared versus separate resource budgets for NTCP2 and SSU2 links to the same peer.
